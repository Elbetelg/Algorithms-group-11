# Project README: Improving Counting Sort via Data Locality

This project implements and evaluates the cache-friendly hybrid sorting algorithm proposed by Shamsed Mahmud, Sardar Anisul Haque, and Nazim Choudhury (2022). The goal is to demonstrate how preprocessing data with Quicksort can improve the performance of Counting Sort by inducing data locality.

## 1. Project Components
The implementation consists of the following Java classes:

* **Main.java**: The central test suite. It reproduces Tables 1, 2, and 3 from the paper.
* **QuickSortModified.java**: Implements "Algorithm 2" (Preprocessing). Single-element sub-arrays are explicitly captured as leaf partitions to ensure correctness.
* **CountingSort.java**: Implements the classic Counting Sort (Algorithm 1).along with a partition-based variant used in the hybrid approach.
* **QuickSort.java**: Standard Quicksort baseline ($T_1$).
* **QuickSortInsertion.java**: Quicksort optimized with Insertion Sort baseline ($T_2$).
* **Partition.java**: Stores partition boundaries (low, high) and value ranges (min, max) used in the hybrid algorithm.

---

## 2. Technical Implementation Details

### The Locality Threshold (C)
In `QuickSortModified.java`, the algorithm stops recursion when the following condition is met:

`(maxValue - minValue + (high - low)) <= C`

We used **$C = 1000$** as per the paper's findings to ensure sub-arrays fit into the CPU cache levels.

### Pivot Selection
All Quicksort variants use **Median-of-Three** pivoting to ensure consistent $O(n \log n)$ preprocessing and prevent performance degradation on sorted inputs.

### Partition-Based Counting Sort (Hybrid Approach)
In the hybrid algorithm, Counting Sort is applied independently to each partition generated by the modified Quicksort. Each partition satisfies the condition:

`(range + size) <= C`

This ensures that the auxiliary arrays (`count[]` and `output[]`) remain small and cache-friendly, improving memory locality and overall performance.

## 3. How to Reproduce Results

### Prerequisites
* Java Development Kit (JDK) 8 or higher.

### Execution Steps
1. Open a terminal in the project folder.
2. Compile all files: `javac *.java`
3. Run the benchmarks: `java Main`

**Note on Methodology:** To ensure reliability, the program performs a **Warm-up Phase** of 10 runs before measuring, and every result printed is an **Average of 20 Trials**.

## 4. Understanding the Output

1. **Table 1 (Locality Test):** Demonstrates that "Sorted" data is ~2-3x faster than "Random" data due to increased cache hits.
2. **Table 2 (Robustness Test):** Evaluates the "Worst Case" ($r \gg n$). Shows the Hybrid Sort's stability when ranges are large. Reports preprocessing time and counting phase time separately, alongside classic Counting Sort alone. Note: the classic sort time remains roughly flat across n = 1000, 2000, 3000 because r = 1,000,000 dominates the runtime — this matches the original paper's results and is expected behaviour.
3. **Table 3 (Final Comparison):** The core result. $T_3$ (Hybrid) should consistently outperform $T_1$ and $T_2$, proving the efficiency of the locality-based approach.

## 5. Developer Notes for Documentation Team
* **Warm-up Loop:** We implemented a 10-iteration warm-up to trigger the JVM's Just-In-Time (JIT) compiler, ensuring the CPU is at peak performance measurements.
* Averaging: NUM_TRIALS = 20 timed trials are averaged per configuration to filter out system noise and produce statistically reliable results.
* **Precision:** We used `System.nanoTime()` divided by `1,000,000.0` to maintain the 3-decimal precision found in the original research tables.
* Hardware Variation: Results will differ from the paper's figures. The paper used a native implementation on a specific Intel Core i7. Our Java/JVM results show the same relative trends but different absolute times focus on performance ratios, not raw numbers.
